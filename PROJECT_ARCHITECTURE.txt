================================================================================
UNIVERSAL FILE READER - PROJECT ARCHITECTURE & LOGIC
================================================================================

A lightweight, modular desktop application for reading any file format.
Built with Python, CustomTkinter, and TkinterDnD.

This document explains the logic, purpose, and implementation of each component.
For reference implementations in other languages, see "PORTING TO OTHER LANGUAGES".

================================================================================
SECTION 1: OVERALL ARCHITECTURE
================================================================================

The application follows a HANDLER-ROUTER-VIEWER pattern (like MVC but for files):

    USER INPUT
         ↓
    [DROP ZONE / FILE BROWSER]  ← Accepts file paths
         ↓
    [ROUTER]  ← Determines file type
         ↓
    [HANDLER]  ← Specializes in file format
         ↓
    [VIEWER]  ← Displays content to user

The key principle: SEPARATION OF CONCERNS.
- Each file type (text, media, generic) has its own handler.
- Each viewer is specialized for its content type.
- The router delegates without knowing details about each handler.

This design makes the app:
  ✓ Extensible — add new file types easily
  ✓ Maintainable — logic is isolated per file type
  ✓ Fast — heavy libraries loaded only when needed
  ✓ Testable — each component has a single responsibility

What it does:
  • Shows last N recently opened files as clickable "chips"
  • Persists history to disk (.file_history.json)
  • Allows quick re-opening of recent files

Why this component:
  • Improves UX by avoiding re-browsing for common files
  • History persists across sessions
  • Compact horizontal layout fits naturally above drop zone

Key functions:
  load_history()      → Read JSON, validate files still exist
  save_history()      → Write JSON (max 8 entries)
  add_to_history()    → Insert at front, remove duplicates, cap at 8

HistoryBar class:
  __init__()          → Setup UI, load history
  _build()            → Create horizontal frame with chips
  refresh()           → Reload from disk and redraw

Storage:
  Location: .file_history.json (one level up from code)
  Format: JSON array of absolute file paths
  Max entries: 8 (configurable _MAX_HISTORY constant)

Persistence logic:
  • On each file open: add_to_history(path) called
  • Path moved to front if already in history
  • Duplicates removed
  • List capped to last 8 items
  • Saved to JSON on disk

Python-specific:
  • Uses json module for persistence
  • os.path.isfile() checks existence before loading


── 4. ROUTER (Business Logic) ────────────────────────────────────────────────

What it does:
  • Matches file path to appropriate handler
  • Updates history
  • Delegates to handler's open_viewer() method

Why this component:
  • Single decision point for file type matching
  • Decouples UI from handler implementations
  • Ensures consistent behavior across the app

Key function:
  route_file(path, parent=None)
    1. Normalize path (cross-platform separators)
    2. Call add_to_history(path)
    3. Try each handler in order: TextHandler → MediaHandler → GenericHandler
    4. First match calls handler.open_viewer(path, parent)
    5. Exit

Handler order (first match wins):
  Priority 1: TextHandler     ← Handles .txt, .pdf, .docx, code files, etc.
  Priority 2: MediaHandler    ← Handles .mp3, .mp4, etc.
  Priority 3: GenericHandler  ← Catch-all for .zip, .exe, etc.

Why this order?
  • Text is most common format
  • Media second most common
  • Generic handles everything else

Error handling:
  • Silent try/except around history update (non-critical)
  • No file existence check here (handlers do it)
  • parent.after(0, ...) ensures viewer runs on main thread

Python-specific:
  • Uses os.path.normpath() for Windows/Unix compatibility
  • parent.after() prevents thread conflicts with Tkinter


── 5. BASE_HANDLER (Abstract Base Class) ────────────────────────────────────

What it does:
  • Defines interface all handlers must implement
  • Provides utility methods (file_ext, human_size)

Why abstract?
  • Enforces consistent API across different handler types
  • Prevents incomplete implementations
  • Acts as documentation

Required methods (subclasses must implement):
  can_handle(path: str) → bool
    Quick check: does this handler support this file?
    MUST NOT do heavy work (no library imports)
    Example: check file extension

  open_viewer(path: str, parent) → None
    Create and show a viewer window for this file
    Called only if can_handle() returned True
    parent: Tk root window (for positioning)

Utility methods (subclasses inherit):
  file_ext(path)      → Return ".ext" (lowercase)
  human_size(path)    → Return "1.5 MB" formatted string

Design philosophy:
  • can_handle() is FAST (just file extension check)
  • Heavy imports deferred to open_viewer()
  • This keeps app startup snappy

Python-specific:
  • Uses ABC (Abstract Base Class) for enforcement
  • @abstractmethod decorator
  • @staticmethod for utility methods


── 6. TEXT_HANDLER (Specialized Handler) ─────────────────────────────────────

What it does:
  • Detects text-readable formats
  • Extracts text from complex formats (PDF, DOCX, RTF)
  • Opens TextViewer window

Supported formats:
  Plain text: .txt, .md, .log, .csv, .json, .yaml
  Code: .py, .js, .ts, .html, .css, .sql, etc.
  Documents: .pdf, .docx, .rtf
  Config: .ini, .cfg, .env, .toml

can_handle() logic:
  1. Check if extension in _TEXT_EXTENSIONS set
  2. If no extension → try opening as text (heuristic)
  3. Heuristic: read 512 bytes, check if >90% are printable ASCII
  4. Return True/False

Why heuristics?
  • Some text files have no extension
  • Better to attempt reading than fail silently
  • 90% threshold avoids false positives on binary files

extract_text() method (called by TextViewer):
  Branches on file extension:
    .pdf    → _read_pdf()    → Uses pdfplumber (lazy import)
    .docx   → _read_docx()   → Uses python-docx (lazy import)
    .rtf    → _read_rtf()    → Uses striprtf (lazy import)
    else    → _read_plain()  → Standard UTF-8 reading

_read_plain() (handles most text):
  1. Try encodings in order: utf-8, utf-8-sig, latin-1, cp1252
  2. Return decoded text
  3. If all fail: decode as latin-1 (never crashes)

_read_pdf():
  1. Open PDF with pdfplumber
  2. Iterate pages
  3. Extract tables as ASCII art
  4. Extract text
  5. Format nicely

_read_docx():
  1. Open DOCX with python-docx
  2. Extract paragraphs and tables
  3. Preserve structure

_read_rtf():
  1. Use striprtf to remove RTF control codes
  2. Return plain text

Why lazy imports?
  • pdfplumber, python-docx, striprtf are heavy dependencies
  • Import only when actually reading that file type
  • App starts instantly even if no PDFs opened yet
  • Users may only read .txt files — why import PDF library?

Python-specific:
  • Lazy imports: done inside function, not at module level
  • Encoding fallback chain: common pattern for text handling
  • os.path.splitext() for extension

SECTION 3: DATA FLOW & SEQUENCES
================================================================================

── Opening a file by drag-and-drop: ──────────────────────────────────────────

  User drops file on DropZone
       ↓
  DropZone._on_drop() triggered
       ↓
  Parse file path from DND_FILES event
       ↓
  DropZone._route_and_open(path) called
       ↓
  router.route_file(path, parent) called
       ↓
  add_to_history(path) updates JSON
       ↓
  Loop through _HANDLERS (in order):
       ├─ TextHandler.can_handle(path)?
       ├─ MediaHandler.can_handle(path)?
       └─ GenericHandler.can_handle(path)? → Always True
       ↓
  handler.open_viewer(path, parent) called
       ↓
  Appropriate viewer (TextViewer/MediaViewer/GenericViewer) created
       ↓
  New Toplevel window appears with content


── Opening a file from history bar: ──────────────────────────────────────────

  User clicks chip in HistoryBar
       ↓
  HistoryBar click callback triggered
       ↓
  router.route_file(path, parent) called
       ↓
  [Same sequence as above from router onwards]


── Opening a file via click-to-browse: ──────────────────────────────────────

  User clicks DropZone
       ↓
  DropZone._on_click() triggered
       ↓
  filedialog.askopenfilename() opens system file picker
       ↓
  User selects file, OK button pressed
       ↓
  DropZone._route_and_open(path) called
       ↓
  [Same sequence as drag-and-drop above]


── App lifecycle: ────────────────────────────────────────────────────────────

  1. main.py executed
  2. App() instantiated
     └─ __init__() called
        └─ _build_ui()
           ├─ DropZone created
           ├─ HistoryBar created + load_history()
           └─ All widgets packed
  3. app.mainloop() starts
  4. Window renders, waits for user input
  5. On user action: event handler fires
  6. New viewer window(s) created as needed
  7. On close: Tk event loop continues
  8. On app close: mainloop() exits, script ends


================================================================================
SECTION 4: ERROR HANDLING STRATEGY
================================================================================

"Fail gracefully" is a design principle throughout:

── File not readable (permission denied, corrupted, etc.): ───────────────────
  • TextHandler catches exceptions in extract_text()
  • Returns error message instead of crashing
  • TextViewer displays: "[Error reading file]"

── Missing heavy library (e.g., pdfplumber not installed): ────────────────────
  • ImportError caught
  • Message shown: "PDF reading not available"
  • Continue with next extraction method or show generic text

── Invalid history file (.file_history.json corrupted): ────────────────────────
  • load_history() wrapped in try/except
  • Returns empty list on error
  • History just won't show, app still works

── File drops on drag-and-drop: ─────────────────────────────────────────────────
  • Threading used to avoid blocking UI
  • Exceptions logged but don't crash

── Media file not playable: ──────────────────────────────────────────────────
  • pygame.mixer error caught
  • Show error label instead of player controls


================================================================================
SECTION 5: DESIGN PATTERNS USED
================================================================================

1. ABSTRACT BASE CLASS (FileHandler, BaseViewer)
   Purpose: Define interface, ensure consistent API
   Benefit: New handlers can be added confidently

2. HANDLER PATTERN (TextHandler, MediaHandler, GenericHandler)
   Purpose: Specialize logic per file type
   Benefit: Easy to add new formats (e.g., ImageHandler)

3. ROUTER / DISPATCHER (router.route_file)
   Purpose: Decouple file selection from viewer creation
   Benefit: Single decision point, testable

4. LAZY LOADING (imports inside functions, not at module level)
   Purpose: Fast startup, only import what's needed
   Benefit: 100ms startup vs 2s if all imports at top

5. COMMAND PATTERN (callbacks in HistoryBar, DropZone)
   Purpose: Pass actions without tight coupling
   Benefit: Flexible, easy to test

6. OBSERVER PATTERN (history_bar.refresh() called from router)
   Purpose: Update UI when state changes
   Benefit: Separation of concerns, reactive updates

7. STRATEGY PATTERN (different extract_text strategies per format)
   Purpose: Switch algorithm based on file type
   Benefit: Clean, extensible text extraction

    public abstract boolean canHandle(String path);
    public abstract void openViewer(String path, Window parent);

public class Router {
    private static final List<FileHandler> handlers = Arrays.asList(
        new TextFileHandler(),
        new MediaFileHandler(),
        new GenericFileHandler()
    );

    public static void routeFile(String path, Window parent) {
        History.add(path);
        for (FileHandler handler : handlers) {
            if (handler.canHandle(path)) {
                Platform.runLater(() -> handler.openViewer(path, parent));
                return;
            }
        }
    }

// src/handlers/router.rs
pub struct Router {

impl Router {

            if handler.can_handle(path) {
                return handler.open_viewer(path);
            }
        }
        Err("No handler found".to_string())
    }

================================================================================
SECTION 7: KEY TAKEAWAYS FOR PORTING
================================================================================

1. ARCHITECTURE IS LANGUAGE-AGNOSTIC
   The handler-router-viewer pattern works in any language.
   Implement the pattern first, libraries second.

2. LAZY LOADING IS UNIVERSAL
   Don't import heavy libraries at startup.
   Import only when needed (JS: dynamic imports, Java: reflection, etc.)

3. SEPARATE CONCERNS
   Text extraction, media playback, routing should be separate.
   This makes each piece easier to test and maintain.

4. UI FRAMEWORK CHOICE MATTERS
   Python has Tkinter (simple, built-in).
   JavaScript has Electron/Tauri (powerful, cross-platform).
   Java has JavaFX (modern, feature-rich).
   Choose based on your needs.

5. HISTORY PERSISTENCE
   Store recent files in JSON (portable across languages).
   Max 8 items to avoid bloat.
   Load at startup, update on every file open.

6. ERROR HANDLING
   Never crash the app. Always show a message instead.
   Catch exceptions at handler level, not globally.
   Let the app continue running gracefully.

7. THREAD SAFETY
   UI updates must happen on main thread.
   Use platform-specific mechanisms:
     Python: Tk.after()
     JS: await in async functions
     Java: SwingUtilities.invokeLater()
     C#: Dispatcher.Invoke()

8. TESTING
   Test each handler independently.
   Mock file system for unit tests.
   Test router logic with fake handlers.

Step 1: Create ImageHandler extending FileHandler
  • can_handle() → check for .jpg, .png, .gif, .bmp, etc.
  • open_viewer() → instantiate ImageViewer

Step 2: Create ImageViewer extending BaseViewer
  • Display image in self.body
  • Add zoom controls (in/out)
  • Add fit-to-window toggle
  • Add rotation buttons

Step 3: Add ImageHandler to _HANDLERS list in router.py
  Insert before GenericHandler (order matters!)

Step 4: Add image libraries to requirements.txt
  PIL (Pillow) for image handling

Step 5: Test!
  Drag an image file to the app.
  It should open in ImageViewer.

That's it! The architecture makes it trivial to add new types.
